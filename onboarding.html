Para devs · onboarding

De clonar el repo a tu primera US trazada

El repo ya viene configurado: las skills, la constitución y OpenSpec viajan con él. Instalas el CLI una vez, y después corres el mismo ciclo para cada User Story.

CLI · Node ≥ 18 Asistente · Claude / Copilot / Cursor Tracker · ClickUp / Jira / md

Parte A

Setup — una vez por máquina

Lo único que se instala por máquina es el CLI. Todo lo demás (skills, constitución, OpenSpec) ya está versionado en el repo.

1

Instala el CLI y autentica el forge

El comando queda como dai. gh (o glab) lo usa dai pr para crear la PR.

$ npm i -g @dforce2055/dai # el CLI, cero dependencias
$ dai --version
$ gh auth login # o: glab auth login
2

Trae el repo ya configurado

Al clonar (o hacer pull) llegan las skills, la constitución y OpenSpec. No corres dai init: eso ya lo hizo quien preparó el repo.

$ git clone <repo> # o: git pull en tu rama base
3

Configura tu .env con tu token

Copia el ejemplo y pega tu token del tracker en la variable correspondiente. Pega solo el token, sin comentarios ni espacios en esa línea. El .env está gitignoreado: no se commitea.

$ cp .env.example .env # y pega tu token del tracker

¿Cómo saco mi token? → Jira · ClickUp

4

Reinicia tu asistente

Para que carguen las skills (/grill-*, /tdd, /link-us, /dai-review) y los comandos /opsx:*. Se cargan al iniciar.

5

Verifica

Verde en skills + el tracker con token = listo para trabajar.

$ dai doctor

Parte B

El ciclo — una vez por cada US

El mismo recorrido para cada historia. Cada paso tiene una skill de IA o un comando que hace el trabajo pesado. QUÉ CÓMO máquina

1

Trae la US y abre el CÓMO

dai trae la User Story del tracker, crea la branch y siembra el link — correcto por construcción, sin tipear el ID a mano. Parte siempre de tu rama base actualizada.

dai link-us <ID> → branch + implements.yaml
2

Diseña con OpenSpec

Exploras la solución y generas la propuesta, el diseño y las tareas — co-localizados con el link, en la misma branch.

/opsx:explore → /opsx:propose
3

El agente implementa con tests primero

Con /opsx:apply, el agente aplica las tareas en rebanadas verticales: un test, el código mínimo, repetir. Escribe el test como spec ejecutable; tú validas.

/opsx:apply · con TDD
4

Verifica la trazabilidad

¿Tu código sigue al día con la US? dai re-deriva el hash de la US viva y compara. Si el QUÉ cambió, te avisa y te dice cómo resincronizar.

dai check → ✅ al día / ⚠️ atrasado
5

Crea la PR

Revisas tu propio código + smoke local, y dai arma la PR desde el template, precargada con la US, el estado del check y los enlaces.

dai pr · luego /dai-review <PR> (la revisa un compañero)
6

Estampa la cobertura y cierra

Al mergear, dai escribe en el tracker quién implementó qué. Después cierra la US: vuelve a la base y borra la branch.

dai stamp → dai done
El <ID> de la US es el identificador del tracker. En ClickUp, de una URL como app.clickup.com/t/9001…/86acme482, el ID es 86acme482. El método y las convenciones del repo están en su CLAUDE.md.

Parte C

Si algo falla

Los tres tropiezos más comunes al empezar, y cómo salir de ellos.

dai link-us tira un error de ByteString

Tu línea del token en el .env tiene un comentario o un espacio de más. Deja solo el token, sin nada después.

No aparecen las skills ni /opsx:*

Reinicia el asistente / IDE. Las skills y los comandos se cargan al iniciar.

dai doctor en rojo

Sigue lo que indica cada línea: casi siempre es el token del tracker o falta reiniciar.
